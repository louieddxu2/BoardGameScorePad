# 軟體規格說明文件 (SSD) — 起始玩家與順位決定器 (v20)

本文件定義「起始玩家與順位決定器」小工具的功能規格、系統互動流程與核心防禦機制。

---

## 1. 核心業務目標：起始玩家與座位/出牌順位決定

本工具的終極目標是為實體桌遊聚會隨機排定**「起始玩家」**與**「隨機出牌順位」**，並將結果無縫回填至計分板。

### A. 參與者類型定義
1. **具名已鎖定玩家 (Type A)**：在推薦名單中找到自己的名字，拉動搖桿鎖定，放開手指後實體化為「氣泡球」，隨後可點擊氣泡選顏色。
2. **匿名玩家 (Type B - 停在原位鎖定)**：推薦名單中沒有自己名字的玩家。**其手指停在原位（不拉動搖桿，位移 $\le 15\text{px}$）維持 3 秒，即自動鎖定為「玩家 n」**。實體化後同樣只能點擊選顏色。不支援就地打字改名，所有改名皆延後至帶回計分板後進行。

### B. Seat Order (座位順序) 填入機制 ─ 欄位對齊
座位順序（實體坐姿方位）與隨機出牌順位是獨立的兩件事。計分板在填入玩家人名時，是依據物理座位的順時針方向排序：
1. **起點定義**：以**第一個按下手指的人（Touch ID 序號最小，或時間戳最早的第一個觸控點位置）**作為座位順序的第 1 位起點（對應計分板最左側第一欄）。
2. **中心點基準**：以 Canvas 畫布的幾何中心點 $(CX, CY)$ 為原心。
3. **順時針排序 (Clockwise Sort)**：
   - 計算所有實體化玩家球相對於中心點的角度 $\theta$。
   - 以第一個按下手指的玩家角度 $\theta_{first}$ 為基準，其餘玩家**依照順時針方向的物理角度遞增排序**。
   - 排定後的座位順序（例如：第一個按的人 -> 順時針下一個 -> 順時針再下一個）即為帶回計分板的玩家排列順序。

### C. Turn Order (出牌順位) 隨機決定與視覺顯示
出牌順位代表遊戲中輪流出牌的順序，由系統隨機決定，並以強大的無障礙與陣營分組視覺呈現在畫面上：
1. **時機點與自動觸發**：當畫面上所有的玩家皆完成鎖定並實體化（$\text{已實體化氣泡數} = \text{計分板設定的人數 (Session Player Count)}$），且手指皆已離開螢幕的**瞬間**，系統立刻啟動閃爍抽籤。
2. **2秒隨機抽籤**：系統在所有已實體化氣泡球之間進行隨機光圈閃爍，動畫總共播放 **2 秒**。
   - *註：即使玩家實體化後完全沒有手動選擇顏色，系統也不會阻塞，時間到依然會自動開始 2 秒閃爍結算。*
3. **隨機順位產生與首家標記**：
   - 動畫播放完畢後，系統隨機排定每位玩家的出牌順位（1, 2, 3, 4...）。
   - 順位 1 即為「起始玩家」，其 `isStarter` 屬性在資料中標記為 `true`。
   - **首家標記（🎯）**：為了避免與計分板中的 Winner 皇冠（👑）產生語意混淆，起始玩家在畫面上**嚴禁使用皇冠標章**。改在該氣泡球上方渲染代表**「首家 / 🎯」**的專屬標章，並在球體周圍播放擴散的光暈波動動畫。
4. **環繞順位數字顯示 (Multi-directional Badges)**：
   - 為了讓圍坐在平板四周不同方位的玩家都能看清，每個氣泡球的**四個方位（3點鐘、6點鐘、9點鐘、12點鐘方向）的外圈邊緣，皆會印上該玩家的順位數字（如 ① ）**。
   - 文字旋轉角度會依循玩家球的 `textRotationDeg` 進行反向修正，以保證任何一側看都是正向。
5. **雙色陣營分組 (Two-Color Team Grouping)**：
   - 順位產生後，系統自動將玩家分為兩大陣營（例如：奇數順位 1、3 為 A 組；偶數順位 2、4 為 B 組）。
   - 氣泡球的外圍光圈會以**雙色（例如：藍色 `#3b82f6` 與橘色 `#f97316`）**進行陣營標記，讓玩家一眼看清隊友與對手。
6. **新觸碰中斷 (Touch Interrupt)**：在 2 秒閃爍期間，若偵測到任何新的 TouchStart 事件（例如玩家欲改色或刪除氣泡），閃爍動畫立刻中斷重設，將控制權還給玩家。所有手指再次離開且人數齊全時，再度重新觸發閃爍。

## 2. 系統互動流程與串接架構

### A. 與推薦引擎串接與內存快取規格
1. **免除高頻讀取資料庫 (No High-Frequency DB Queries)**：
   - **資料庫讀取時機**：為確保 Canvas 物理動畫的流暢度與避免 I/O 阻塞，**推薦引擎對資料庫 (IndexedDB) 的讀取與計算僅會在「三個特定時間點」觸發**：
     1. 剛進入「順位決定器」視窗**初始化**時。
     2. 當有玩家**成功鎖定 (LOCKED)** 身分時（更新 `knownPlayerIds`）。
     3. 當有實體化玩家被**刪除 (❌)** 時（從 `knownPlayerIds` 移除）。
   - **內存快取過濾**：當玩家進行「快速抬起再按壓 (Tap-to-Refresh)」以刷新名字時，系統**僅在記憶體已載入的候選名單中過濾 `skippedPlayerIds` 並翻頁**，完全不需要發起任何 IndexedDB 查詢。
2. **已鎖定玩家的「動態局部頂替機制」 (避免選項閃爍跳動)**：
   - 當玩家 A 鎖定了名字 `小明`，系統掃描其他正在 `CHOOSING` 狀態的搖桿。
   - 若其選項中也含有 `小明`，**僅將 `小明` 這一個氣泡進行局部替換（淡出淡入動畫）**，替換的新名字由推薦引擎重新計算（自動避開該搖桿現有的其他三個選項，且避開已被鎖定的所有人，並同時避開該觸控點本身的 `skippedPlayerIds`），其餘三個選項保持完全靜止，防止選項閃爍跳動。
3. **顏色選單之產生與預設分配的一致性**：
   - 顏色選單中顯示的候選顏色，是由推薦引擎針對該玩家計算並**排序後**的顏色清單。這份清單本身就已經完全包含了玩家的歷史預測偏好色與該遊戲模板的主題色。
   - 如果玩家沒有手動點選顏色，系統會自動指派該推薦顏色清單中的「首位顏色（第一名）」作為他的預設色。
4. **顏色動態頂替與排他機制 (Dynamic Color Replacement)**：
   - 當顏色 $C$ 被選定或指派後，此顏色會被加入已佔用名單（`usedColors`）。
   - 其他所有仍在編輯（`COLOR_PICKING`）的玩家，其調色盤選單中的 $C$ 會自動收起（隱藏），並在原地翻轉替換為其推薦清單中的「下一個未被佔用」的推薦色。這能確保每個玩家的調色盤按鈕永遠是 100% 可選的有效顏色。

### B. 雙人同時鎖定同名字之競態條件 (Race Condition)
* 當兩位玩家幾乎在同一時間拖曳鎖定同一個名字時，系統以 **100% 讀條成功（達 100% 瞬間）的微秒級時間戳** 作為判定基準。
* 較快達到的玩家 A 獲得身分，寫入 `lockedPlayerIds`；慢了一步的玩家 B，其搖桿讀條應**立刻中斷並歸零**，且其對應的選項氣泡球應立刻播放翻轉動畫，**自動頂替**為下一位推薦名字。

## 3. 手指反覆抬起與取消 (Touch Lift & Cancel) 與選單收合

### A. 未鎖定前抬起手指：極速刷新 (Tap-to-Refresh) 規格
當玩家在 `CHOOSING` 狀態下抬起手指（未滿 750ms），該搖桿與選項氣泡會淡出銷毀。系統採用**「空間優先、時間在後」**的兩階段篩選，並在每個觸控點維護一個已跳過 ID 的輕量集合，以因應動態變化的推薦名單：

* **狀態變數定義**：對於畫面上每個活躍的觸控點，系統維護一個簡單的字串集合：
  - `skippedPlayerIds: Set<string>`（初始為空 `new Set()`，僅儲存此座位已被使用者跳過的玩家 ID）
* **判定步驟 (空間優先機制)**：
  1. **第一關：空間距離檢查 (Spatial Check)**
     - 計算新按壓起點與同座位上一次釋放點的直線距離 $\Delta d$。
     - **若 $\Delta d \ge 50\text{px}$**：判定為位置重設（更換座位或非同人操作）。**系統立刻清空已跳過名單：`skippedPlayerIds.clear()`**，重新回歸第一順位推薦。
     - **若 $\Delta d < 50\text{px}$**：判定在同座位區域，進入第二關時間檢查。
  2. **第二關：時間差檢查 (Temporal Check)**
     - 計算新按壓與上一次釋放的時間差 $\Delta t$。
     - **若 $\Delta t < 500\text{ms}$**：判定為連續刷新。**系統將上一次顯示的 4 個玩家 ID 加入 `skippedPlayerIds` 集合**，下次推薦引擎在內存計算時自動過濾排除這些 ID，以顯示全新一批名單。支援多次連續刷新。
     - **若 $\Delta t \ge 1000\text{ms}$ (或介於 500ms 至 1000ms 之間)**：判定為非連續誤觸。**系統清空已跳過名單：`skippedPlayerIds.clear()`**，還原最優預測。
* **多次連續刷新支援 (Chained Refreshes)**：
  - 只要玩家反覆滿足 $\Delta d < 50\text{px}$ 且 $\Delta t < 500\text{ms}$，`skippedPlayerIds` 就會持續累加，實現多次連續翻頁。
  - 由於此機制記錄的是確切被跳過的 ID，因此即使其他玩家在此期間鎖定了新角色，導致全域推薦順序重新計算，系統依然能絕對精準地過濾掉該座位已經看過的名字。

### B. 實體化後的編輯、重置與選單收合機制
為避免誤觸直接刪除玩家，並維持畫面整潔，調色盤與編輯選單的開啟與收合規則如下：
1. **開啟選單**：點擊實體化氣泡。系統亮出調色盤選單，並在氣泡外圍右上角顯示「❌ 刪除」按鈕。不提供任何改名按鈕或打字輸入。
2. **收合確認機制 (僅以下二種途徑，點擊背景空白不收合)**：
   - **點擊氣泡球名字本身（中心區域）**：由於「❌」顯示在氣泡外圍右上角，玩家點選氣泡球名字本身（中心區域），即視為確認並**收合調色盤選單**，不會有誤觸外側 ❌ 的風險。
   - **閃爍結算強制收合**：當條件滿足，啟動 2 秒閃爍抽籤的瞬間，**系統會強制淡出並收合畫面上所有尚未關閉的調色盤選單**，確保抽籤畫面最乾淨。
   - *註：為防止干擾其他玩家的多點觸控操作，點擊畫布背景空白處、或者點擊其他人的觸控點，均「不會」收合當前已打開的調色盤選單。*

### C. 「開始遊戲」與「再來一次」結算控制流
1. **氣泡向邊緣退避 (Outward Retreat)**：
   - 2 秒隨機閃爍抽籤完畢的瞬間，系統會計算每個氣泡球與畫布中心點 $(CX, CY)$ 的相對方向。
   - 所有已實體化的氣泡球會**自動且均勻地朝向畫布邊緣方向退讓（向外推移）**，使畫布中心騰出一個直徑約 $200\text{px}$ 的乾淨圓形空白區域。
   - **目的**：這能徹底避免玩家氣泡球與中央控制按鈕發生遮擋，且能為氣泡球預留更寬敞的空間，方便玩家點擊進行後續的顏色微調。
2. **中央雙控制按鈕**：
   - 在畫布騰出的正中央空白區域，系統會渲染兩個醒目的控制按鈕：
     - **「開始遊戲」按鈕**：確認結果並將座位順序、各玩家選定的顏色以及起始玩家標記帶回計分板並關閉彈窗。
     - **「再來一次」按鈕**：保留當前的玩家名字與代表色，**立刻重置並重跑一次 2 秒的隨機閃爍抽籤**，重新分配出牌順位、起始玩家與分組。
3. **結算後調整代表色**：在使用者最終點擊「開始遊戲」按鈕之前，調色盤選單依然完全可用。

## 4. 系統健壯性與物理細節 (Robustness & Physics Specs)

### A. 物理引擎細邊界與公式 (Physics & Interaction Engine Specs)
1. **彈力與摩擦力模擬 (Spring and Friction)**：
   - 搖桿選項受彈力公式約束：$vx += (targetX - x) \times SPRING\_K$（預設 $SPRING\_K = 0.08$）。
   - 全局施加摩擦力衰減：$vx \times= FRICTION$（預設 $FRICTION = 0.82$）。
2. **防遮擋排斥力 (Anti-Occlusion Exclusion Zones)**：
   - **手指禁區**：以觸碰錨點為中心，設 $FINGER\_EXCLUSION\_RADIUS = 50\text{px}$ 的排除禁區。
   - **實體氣泡禁區**：已實體化的氣泡周圍設 $OTHER\_EXCLUSION\_RADIUS = 110\text{px}$ 的排除半徑。
   - **邊界排斥**：當氣泡球靠近邊緣時，距離小於 $WALL\_REPULSION\_DIST$ 時施加反向力道，防範球體穿模。
3. **指尖朝向預測 (Capacitive Touch Orientation - Ellipse Prediction)**：
   - 當 $touch.radiusX$ 與 $touch.radiusY$ 大於 0 且兩者比例大於 1.1 時，計算接觸面橢圓形主軸角度。
   - 利用該角度修正「指尖朝向（Human Direction）」，使搖桿周圍的 4 個氣泡以最佳視角對齊玩家本人的生理方位。若硬體不支援（回傳 0），則自動 fallback 為「Canvas 中心點指向觸碰點的外側方向」。

### B. 物理座標與 Viewport 的響應式防線 (Normalized Coordinates)
* 物理引擎底層使用**正規化座標 (Normalized Coordinates，範圍 $0.0 \sim 1.0$)** 來記錄氣泡球與觸碰點的位置，而非寫死像素 (px) 座標。
* 當 `window.resize`（如實體鍵盤關閉、瀏覽器工具列隱藏、網址列伸縮）觸發時，Canvas 重新縮放，物理引擎依據新幾何大小按比例還原氣泡球位置 $(Px = width \times NormalizedX)$，避免球體被擠壓出畫布。

### C. 效能防線與 OOM 防禦
* **性能要求**：必須使用 HTML5 Canvas 2D API 進行每一影格 (Frame) 的高頻渲染，嚴禁每秒銷毀並重建 SVG DOM。
* **事件攔截與系統手勢防禦**：
  - 在 Canvas 容器上套用 `touch-action: none; -webkit-user-select: none; user-select: none;`。
  - 將所有 touchstart 與 touchmove 事件設定為 `{ passive: false }`，且在處理器內顯式調用 `e.preventDefault()` 以完全封鎖 iOS 系統級三指編輯、四指切換與橡皮筋回彈手勢。
