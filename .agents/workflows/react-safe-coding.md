# React 安全編碼與專案規範防呆 SOP (React Safe Coding Guide)

> 📌 **適用場景**：在進行任何 React 組件開發、修改、Bug 修復或進行 i18n 串接之前，必須先閱讀並對照此清單。

---

## 🛡️ 核心三大防呆檢驗清單 (Pre-flight Checklist)

在每次提交程式碼或執行 `git push` 前，AI 必須逐項自檢以下三條核心紅線：

### 🚨 紅線一：React Hooks 依賴與死循環安全 (Hook & Effect Safety)
* **規則描述**：撰寫 `useEffect`, `useMemo` 或 `useCallback` 的依賴陣列時，必須對每一個放入的變數進行「參照穩定性」審查。
* **嚴禁行為**：
  * **嚴禁**將自訂 Hook 返回的未經過記憶化 (Unmemoized) 的函數（例如本專案自訂的 `t` 翻譯函數）放入依賴陣列中。
  * **嚴禁**將在 Render 過程中即時宣告的物件、陣列或 inline 函數放入依賴陣列。
* **正確做法**：
  * 對於 `useEffect`，只監聽真正需要觸發副作用的「狀態狀態」（例如 `isOpen`, `activeId`），無關的工具函數、輔助參照應一律移出依賴陣列。
  * 如果自訂的函數或物件非得放入依賴項，必須確保該變數使用了 `useCallback` 或 `useMemo` 鎖定了記憶體參照，或是從根源重新設計資料流。

### 🚨 紅線二：零硬編碼中文與 i18n 標準流程 (Strict i18n Boundaries)
* **規則描述**：本專案為離線優先 PWA，多語言支援是核心架構。
* **嚴禁行為**：
  * **嚴禁**在 `src/i18n/` 以外的任何 `.tsx` / `.ts` 組件或檔案中寫下任何硬編碼中文字串（如按鈕 Label、彈窗標題、Toast 提示等）。
* **正確做法**：
  * 必須嚴格執行 `/.agents/workflows/add-i18n-string.md` 的標準流程。
  * **「先有翻譯 key，後有代碼引用」**：必須先在 `src/i18n/` 下的對應模組或全新模組補齊 `zh-TW` 與 `en` 翻譯對照表，才能在 UI 組件中透過 Hook 調用。

### 🚨 紅線三：保守開發與專案沙盒邊界 (Conservative Boundary & Revert First)
* **規則描述**：修改 Bug 時專注於問題本身，絕對不隨意擴大影響範圍。
* **嚴禁行為**：
  * **嚴禁自作聰明**：當遇到連線或開發瓶頸時，嚴禁在未經使用者授權的情況下引入外部第三方 CDN、Raw GitHub URL 或未授權的連網地址。
  * **嚴禁夾帶重構**：當某個局部 Bug 是由於新代碼引起時，嚴禁提議重構原本運作良好、健全的既有代碼（例如重構全域 i18n 架構）來掩蓋單一 Bug。
* **正確做法**：
  * 專注於修復受影響的局部組件。
  * 本地開發（Local Dev）的降級機制（Fallback）應純粹保持在本地 `public/` 的靜態 Mock JSON 中，維持完美的本機沙盒防線。

---

## 📈 違規懲罰與還原機制
1. 如果自動化掃描腳本 `verify.ps1` 報出任何 `Hardcoded Chinese UI string`，AI 必須立刻無條件修復為 i18n Key，不得提交。
2. 若程式碼引發了高頻率渲染死循環，或造成了 API/資料庫查詢暴增，AI 必須立刻使用 `git reset --hard` 還原至穩定版本，並向使用者進行技術究責。
